Phase 1: Structural Mapping
Architectural Pattern: Layered Monolith
The system follows a classic three-tier layered architecture (not Clean Architecture or Hexagonal — there is no dependency inversion at the boundary level). The layers are well-named and consistently organized:


Routers (HTTP interface)
  └── Services (business logic)
        └── DB Models (persistence via SQLAlchemy ORM)
              └── PostgreSQL
The LLM abstraction (llm_provider.py) is a clean Strategy pattern with a factory, which is a highlight of the codebase.

Phase 2: Vulnerability & Risk Assessment
SEVERITY: CRITICAL
[SEC-1] Live credentials committed to .env

backend/.env contains production secrets that should never be in a file that could enter version control:


SECRET_KEY=67aac32a...          ← JWT signing key (anyone who has this can forge tokens)
GOOGLE_CLIENT_SECRET=GOCSPX-…  ← Google OAuth secret (can impersonate your OAuth app)
openrouter_api_key=sk-or-v1-…  ← Will incur billing charges if abused
gemini_api_key=AIzaSyDMg…      ← Same
DATABASE_URL=postgres:root@…   ← "root" password on the database
Action required before any commit is pushed: Rotate all of these credentials immediately. Add .env to .gitignore (the backend has its own .gitignore — confirm it excludes .env).

[SEC-2] Rate limiter is trivially bypassable via header spoofing

backend/app/core/limiter.py:8-10:


forwarded = request.headers.get("X-Real-IP")
return forwarded if forwarded else get_remote_address(request)
Any client can send X-Real-IP: 1.2.3.4 in their request. Since there is no nginx upstream that strips/overwrites the header, the effective rate limit per attacker is infinite. The header should only be trusted if you can guarantee it was set by your proxy, not by the client.

[SEC-3] POST /auth/refresh has no rate limit

Confirmed by reading backend/app/routers/auth.py:124-142 — the /auth/refresh handler has no @limiter.limit() decorator. An attacker who obtains a refresh token can call this endpoint in a tight loop to extend their access indefinitely without triggering any throttle.

[SEC-4] Refresh tokens are non-revocable (stateless JWT)

There is no token blacklist, no server-side session table, and no jti (JWT ID) tracking. A stolen refresh token (7-day TTL) gives an attacker full authenticated access with no way to invalidate it short of rotating SECRET_KEY, which would log out all users.

SEVERITY: HIGH
[SEC-5] Synchronous HTTP client blocks the async event loop

backend/app/services/llm_provider.py:71:


self._client = httpx.Client(timeout=45.0)   # ← synchronous
FastAPI runs on an async event loop (uvicorn/asyncio). Calling httpx.Client.post() from a sync function that FastAPI runs in a threadpool is acceptable only if the route handler is def not async def. All three router handlers that call generate_review_text are def (not async def), so FastAPI correctly runs them in a threadpool. This is currently safe, but fragile — any future async def migration will silently reintroduce event loop blocking.

[PERF-1] /reviews/history has no pagination

backend/app/routers/reviews.py:70-92: The history endpoint fetches all GeneratedReview rows for a user and builds the full list in memory before returning. A power user with thousands of entries will cause a large DB read and a large JSON payload on every page load.

[ARCH-1] Missing backend endpoint called by the frontend

frontend/src/api/client.js:137-143:


export const updateUserProfile = async (profileData) => {
    const { data } = await api.patch("/auth/me", profileData);
There is no PATCH /auth/me route in backend/app/routers/auth.py. The EditBusinessPage will always return a 405 or 404 silently. This is a broken feature.

[ARCH-2] Business logic embedded in the router

backend/app/routers/auth.py:89-113 — the Google auth router contains a 25-line username-generation loop with retry logic. This belongs in services/auth.py, not in the HTTP handler. Routers should only translate HTTP ↔ domain; they should not contain loops, conditionals over domain state, or raw DB queries.

SEVERITY: MEDIUM
[SEC-6] JWT tokens stored in localStorage (XSS risk)

frontend/src/api/client.js:11 and frontend/src/context/AuthContext.jsx:12: Both the access and refresh tokens are stored in localStorage, which is readable by any JavaScript on the page. A single XSS vulnerability (in a dependency, a third-party script, or future ad/chat widget) would exfiltrate both tokens. httpOnly cookies are the standard mitigation.

[PERF-2] Module-level LLM provider singleton with no lazy initialization

backend/app/services/review.py:10:


_provider = get_provider()   # called at import time
This initializes the provider (and validates the API key) the moment the module is first imported — before the application even starts serving requests. A misconfigured .env crashes the server at boot. The provider should be lazy-initialized or use FastAPI's dependency injection system, which also enables proper testing isolation.

[SEC-7] CORS allow_methods=["*"] is overly permissive

backend/app/main.py:37: The application only needs GET, POST, and PATCH. Allowing all methods (DELETE, PUT, OPTIONS, HEAD) unnecessarily expands the attack surface on browsers that honor CORS preflight.

[ARCH-3] ARRAY column creates an implicit PostgreSQL dependency

backend/app/db/models.py:17:


seo_keyword = Column(ARRAY(String), ...)
sqlalchemy.dialects.postgresql.ARRAY only works with PostgreSQL. The test suite uses an SQLite in-memory DB (via conftest.py). This will silently fail or require patching when running tests unless the conftest overrides the engine — which needs verification.

[ARCH-4] Dev-only schema creation in production code path

backend/app/main.py:22-24:


if settings.debug:
    Base.metadata.create_all(bind=engine)
The guard is settings.debug, which defaults to False and is fine. But the risk is that a misconfigured production deployment with DEBUG=true will bypass Alembic migrations entirely and create tables without migration tracking, leading to schema drift.

SEVERITY: LOW
[SEC-8] Prompt injection guard is keyword-based and easily bypassed

backend/app/services/review.py:129-143: The injection detection checks for exact substrings ("ignore", "act as", etc.). An attacker can trivially bypass this with "1gn0re", "ıgnore" (Turkish ı), spacing, or encoding. The guard provides false confidence. The real mitigation is prompt structure (the system prompt already separates user input into a clearly labeled block, which is the correct defense).

[INFRA-1] Ollama exposed on 0.0.0.0 in the VPS setup script

scripts/vps_setup.sh:22-27: The script comments out the firewall rule (ufw allow from ...) and marks it [MANUAL], making it easy to ship with Ollama publicly accessible on port 11434. The script should apply the firewall rule automatically and fail loudly if YOUR_APP_SERVER_IP is not substituted.

Phase 3: Qualitative Analysis
Coupling & Cohesion
The reviews.py router touches settings directly to re-validate the LLM config, which the llm_provider factory already guards. This is double-checking the same invariant in two places — remove the config checks from the router and let the service layer raise a 503 if the provider isn't configured.

The UserBusiness model is doing double duty: it is both the auth entity (credentials, google_id) and the business profile (seo_keyword, business_desc, review_link). As the product grows, splitting this into User + BusinessProfile would allow users to manage multiple business profiles — a foreseeable feature.

DRY / AHA Balance
The codebase is appropriately DRY. The _review_url() helper is defined in two routers (reviews.py:14 and public_review.py:17) with slightly different logic — this should be consolidated into a single service-layer utility.

The fallback review pool in review.py and the prompt template are well-separated and easy to maintain individually. No premature abstraction detected.

Architecture Decision Records (ADRs)
ADR-1: Add PATCH /auth/me and split User ↔ BusinessProfile
Problem: The frontend already calls PATCH /auth/me but the route does not exist. Additionally, UserBusiness conflates authentication identity with business profile data.

Recommendation: Add the PATCH /auth/me endpoint immediately (unblocks EditBusinessPage). As a longer-term change, split UserBusiness into User (auth fields) and BusinessProfile (business fields) via an Alembic migration. This enables multi-profile support and cleaner authorization logic.

ROI: High — unblocks a user-visible feature today, and the model split pays dividends every time you add a new business-profile field.

ADR-2: Move to httpOnly cookie auth (or add a token revocation table)
Problem: localStorage-stored JWTs are vulnerable to XSS and cannot be revoked server-side.

Recommendation (Option A — easier): Add a refresh_token_jti column to UserBusiness (or a separate sessions table). On logout or suspicious activity, set the jti to a new UUID, invalidating all outstanding refresh tokens for that user.

Recommendation (Option B — more secure): Issue tokens via Set-Cookie: httpOnly; Secure; SameSite=Strict from the backend, removing tokens from JavaScript's reach entirely. This requires CORS adjustments and frontend changes, but eliminates the XSS vector class.

ROI: High — the current setup means a single npm dependency with an XSS vulnerability would expose all users' sessions.

ADR-3: Migrate LLM provider calls to async (or make the pattern explicit)
Problem: httpx.Client (sync) is used inside def route handlers, which FastAPI correctly runs in a threadpool. This is safe today but the implicit contract is fragile — a future async def migration will silently block the event loop.

Recommendation: Replace httpx.Client with httpx.AsyncClient in all providers, convert generate() to async def generate(), and update route handlers to async def. FastAPI handles the thread pool automatically for sync functions; using async throughout is explicit and eliminates the footgun.

ROI: Medium — prevents a silent performance regression during any future async migration and enables proper connection pooling.
